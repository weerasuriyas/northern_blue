System overview

A single Next.js 15 app serves both the customer storefront and the admin panel. A local MySQL database acts as the read‑cache and extension layer; Shopify remains the source of truth for commerce data and stays in sync via webhooks.

Sync architecture

Shopify is the source of truth. The DB is a cache + extension layer. Reads always hit the DB (fast, no rate limits); writes go to Shopify first then update the DB to match.

Bulk sync (boot / on demand)

# All tables
npm run db:sync

# Single table
npm run db:sync -- orders
npm run db:sync -- customers
npm run db:sync -- inventory
npm run db:sync -- discounts
db/sync.js calls the Shopify Admin REST API (2024‑10), paginates, and feeds rows through the same upsert helpers in lib/sync.js. Idempotent — safe to re-run.

Read / write contract

ReadsAlways hit MySQL. Never rate‑limited. lib/shopify.js reshapes rows into the Shopify Storefront API form so storefront components don't change when real Shopify is wired.
WritesShopify first → then update DB. Webhook arrives moments later and overwrites with authoritative state.

Wiring up Shopify (3 file swaps)

  1. lib/shopify.js — uncomment GraphQL client, swap DB queries for Storefront API calls.
  2. lib/cart.js — replace localStorage stubs with Shopify Cart API mutations.
  3. lib/sync.js + db/sync.js — already wired; just set SHOPIFY_ADMIN_API_TOKEN and run npm run db:sync.

Commands

docker compose up -d        # start MySQL on :3306
docker compose down         # stop (data persists in volume)
docker compose down -v      # stop + wipe data

npm install
npm run dev                 # Next.js dev (Turbopack)
npm run build               # production build
npm run start               # production server

npm test                    # Vitest watch
npm run test:run            # Vitest run

npm run db:seed             # seed dev data (idempotent)
npm run db:sync             # bulk pull from Shopify Admin API
